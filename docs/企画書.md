# 企画書：PROXY CHAMPIONS（プロキシ・チャンピオンズ）

> 育成 × 全国大会バトルRPG
> 作成日: 2026-06-30 / タイトル確定: 2026-07-04 / 戦闘モデル確定: 2026-07-04 / ステータス: コンセプト固め完了・MVP設計前

> **タイトル由来**：proxy＝代理・分身。自分の分身が代理で全国大会を戦う＝「育てて送り出し、後で観に行く」非同期観戦型の核をそのまま表す。略称 **PROXY**。呪文/攻撃名は英語・元素ベースの一般名で統一（IP対策・4章）。

## 1. コンセプト

- **育てたキャラを全国大会に登録し、トーナメントで勝ち上がる**育成 × 非同期観戦バトル。
- バトルは **RPG方式（自動計算）**。プレイヤーは結果を**指定時刻以降にいつでも確認**できる。
- **演出はドラクエ3風の地味さ**。黒背景・魔法演出は最低限・ダメージ時に画面が揺れる程度。派手な演出は要らない。
- 一番の強み = **「非同期 × 観戦型」**。プレイヤーを拘束しない。育てて、登録して、結果を見に行く。忙しい人にも刺さる。

### コア体験の一言
> 毎日ちょっと育てて、自分好みのキャラを大会に送り出し、後で結果を観に行く。

## 3. 育成（本企画の本丸）

### 3.1 基本思想
- **「時間 or お金 → 強さ」** を採用（F2Pの王道）。時間のある人が強くていい。時間のない人は**課金で時間を買う**。
- ただし強さの最終的な差は **「ビルドの巧さ」** で出す（誰でも時間をかければ天井に届く＝俺つえー）。
- 育成の主語は **「選手（自分の分身）」**。監督/コーチ型ではなく「自分が頑張る → 分身が強くなる」のシンプルな power fantasy（ドラクエウォーク型）。

### 3.2 2つのリソース（ドラクエ式）
| リソース | 用途 | 軸 |
|---|---|---|
| **経験値** | レベルアップ → ステ振り | **縦**（地力。収束する） |
| **ゴールド** | 武器・防具の購入 | **横**（ビルドの個性・型） |

### 3.3 経験値の入手チャネル（プラグイン式・複数併用）
「行動 → 正規化された経験値」という**枠**だけ作り、後から差し込めるようにする。

- **主軸：派遣ダンジョン**（後述 3.3.1）← バトルエンジン再利用・世界観と地続き・ビルド検証も兼ねる
- **任意ルート（間口拡張）**：歩数 / パズル / オセロ等のミニゲーム / **動画広告視聴**（最低労力 + 収益を兼ねる）
- **原則：報酬は「投資した時間」に比例する（＋回復薬課金で時間を買える）**。**日次上限は設けない**。時間・お金をかけた人ほど強くなる。伸びの鈍化はソフトキャップ曲線（3.4）だけが担う。
- **報酬の仕分け**：縦（強さ＝経験値・ゴールド）はどのチャネルで増えてもよい。横／見た目／回復薬などはチャネル固有に配ってよい（賑やかし）。

### 3.3.1 派遣ダンジョン（主軸の具体設計）
- **派遣**：キャラをダンジョンに送り、**指定時間まで自動で連戦**する。アプリは閉じてよい（サーバーが計算・記録＝非同期・拘束しない）。
- **体力（HP）ループ**：バトルでダメージ → 体力が減る。指定時間まで戦い続け、**体力0で強制帰還**（そこまでの報酬は持ち帰る／残り時間は取り逃す）。
- **帰還後**：体力は**時間で自然回復**。体力1以上なら**即・再派遣**できる。
- **回復薬**：体力を一気に回復。入手＝ダンジョンドロップ / ミニゲーム / **将来は課金ショップ**（＝時間を買う課金。多く回す＝速く強くなる、に効く）。
- **ダンジョンの多様性**：タイプ・難易度・推奨潜航時間が様々。**長時間×高難度＝取り分大だが途中でやられやすい／簡単＝安全だが取り分小**。「自分のスケジュール × リスク許容度」でダンジョンを選ぶのが戦略（＝放置ゲーの意思決定）。
- **即時オートも軽く用意**（1戦だけ即実行）＝練習試合(5.3)を兼ね、ビルド検証のテンポを上げる。
- ⚠️ **体力（HP）は「長く潜れる＝uptime」にも効くため価値が二重**（戦闘の生存＋派遣の持久）。**戦闘式で"最強ステ"にしない**よう要調整（3.5 のハズレ・最強ステを作らない、に接続）。

### 3.4 成長曲線：ソフトキャップ（= RPGの経験値曲線）
- 序盤はグングン、レベルが上がるほど必要経験値が増えて伸びが鈍る。**「ずっと上がるが、だんだん寝る」**。上位に行くほど成長が鈍る、というありふれた体感。
- **目的は"公平化・収束"ではない**。狙いは2つだけ：① **数値インフレ（桁）の抑制**、② **上位の伸び鈍化**という納得感のある手触り。**「時間・お金をかけた人ほど強い」は維持したまま、青天井の暴走だけを抑える**。
- **日次上限は設けない**（時間・金を注いだ分だけ伸びる）。ブレーキは**この曲線のみ**。
- 後発でも、**より多くの時間（or お金）を投じれば先行者に追いつき追い越せる**（累積投資＝強さ）。伸びの鈍化は全員共通なので、投資量の大小がそのまま強さ順になる。

### 3.5 ステ振り（独自性の核）
- レベルアップ時、**ポイントを自分で自由配分**（例:10ポイント →「力9素早1」「力1素早10」等、好きに）。
- 観戦型ゆえ**プレイヤーの戦略入力は実質これ（＋装備・編成）**＝**ゲームプレイ本体**（※非魔法の特技は持たせない。魔法呪文＝攻撃/回復/補助はある）。
- ⚠️ **自由なだけだと最適解に収束し独自性が死ぬ**。本物の独自性のために必須：
  1. **三すくみ（カウンター関係）** … 唯一の最強を作らない
  2. **ハズレ・ステを作らない** … 全ステに役割
  3. **一点振りに逓減**（任意）
- **振り直し（リスペック）は必須**。永続キャラ × メタ変動に対応。**ゴールド消費**で実行（ゴールドのシンクも兼ねる）。

### 3.5.1 ステータス定義（MVP確定）
- **基本ステ（5種）＝キャラの生まれ持った特性**（レベルアップで得た能力ポイントを1プールから配分。**5ステは同じスケールに揃える**）：
  - **体力（フィジカル）** → 最大HPを決める（例：最大HP＝体力×10）
  - **魔力（マジック）** → 最大MP（例：魔力×5）＋魔法の威力・状態異常の成功率＋**受ける魔法ダメージの軽減**（魔力 vs 魔力）。＝magic-mightで**攻防兼任**
  - **力（パワー）** → 物理攻撃
  - **素早さ（スピード）** → 行動順・物理回避
  - **精神（メンタル）** → **状態異常耐性（のみ）**（相手の魔力 vs 精神）。※**魔法ダメージは精神では防げない**（軽減は魔力の役目）
- **HP/MPは派生値**：スケールの違うHP/MPを直接振らず、**体力・魔力から倍率で算出**（基本5ステを同スケールに保てる／倍率は仮・調整対象）。→ `最大HP＝体力×10`なら魔法の「最大HP割合ダメージ」が体力投資に比例＝体力全振りのブレーキも自然に効く。
- **物理防御は"ステ"でなく"装備（鎧・盾）"が担う**（3.6）。理由＝物理ダメージの軽減は生まれ持った才能でなく「**着るもの**」の役割 → **守備ステは廃止**。
- **攻撃力の出どころ**：物理＝**力 ×武器** ／ 魔法＝**魔力 ×呪文ライン**（力↔魔力＝攻撃ステ、武器↔呪文ライン＝型/専門）。
- **役割分化は呪文ラインが担う（維持）**：魔力は**全魔法に効く一般倍率**（magic-might）だが、**どの呪文をどのTierまで持てるか＝専門性は呪文ライン投資**（キュアー線＝僧侶／ファイア線＝魔法使い、1プールのトレードオフ）。魔力だけ上げてもライン未投資なら強い呪文は撃てない → 分化は崩れない。
- **脅威と防御（3種の脅威を別々の受けで防ぐ）**：

  | 脅威 | 攻撃側 | 防御側 |
  |---|---|---|
  | 物理ダメージ | 力（＋武器） | **鎧（装備）**＋素早さで回避 |
  | 魔法ダメージ（ファイアー） | 魔力（＋Fireライン） | **魔力**（`基本＋術者魔力−受け手魔力`・下限あり） |
  | 状態異常（スリープ） | 魔力（成功率）＋眠りライン | **精神**（術者魔力 vs 精神・floor/ceiling） |

  ＋ 体力→HP（生命力プール）、魔力→MP（呪文の燃料）。
- **★単一ステで魔法無敵にならない**：精神だけ上げても**スリープは防げるがファイアーで大ダメージ**／魔力だけ上げても**ファイアーは軽減できるがスリープにかかる** → 魔法を封じるには**魔力も精神も**要る。魔力は攻防兼任なので「物理脳筋（低魔力）は魔法に溶ける／魔法使い同士はファイアーが基本値付近に収束」も自然に成立。
- **数式（仮）**：魔法ダメージ＝`Fire基本＋術者魔力−受け手魔力`（**下限＝基本の数%等**）。状態異常成功＝魔力(攻) vs 精神(防) の相対（0/100%にしない）。
- **1プール制＝トレードオフが核**：ある役割に振る＝別が下がる（役割分化の担保）。
- **2経済の均衡**：XP（縦）＝体力/魔力/力/素早さ/精神・呪文修行 ／ ゴールド（横）＝武器＋**鎧（物理防御）**。両通貨が「強さ」に要る。
- **相性（戦闘式から創発・ソフト60:40）**：攻撃は物理/魔法の2タイプ、防御は**鎧（物理）／精神（魔法）**の2系。相手を選べない非同期ゆえ「**鎧を固めるか精神を上げるか**」の読み合い（鎧偏重＝魔法に弱い／精神偏重＝物理に弱い／素早さは物理を躱すが魔法は躱せない）。
- **HP全振り対策**：魔法の一部を**最大HP割合ダメージ**にする。
- **名称**：HP採用（LPは任意表記）。すべてRPG汎用語＝**IP上の問題なし**。
- **特技（非魔法の技）は無し**。ただし**呪文（攻撃・回復・状態異常・バフ）はある**（DQ流に言えば特技≠呪文の切り分け）。呪文は少数の厳選リスト。

### 3.5.2 魔法の習得（呪文ライン別の"修行"式）
- 各**呪文ライン**（攻撃 / 回復 / 状態異常 / バフ）に、レベルアップのポイントを振って**修行値**を上げる（例：キュアー +3）。
- **威力は投資量で連続アップ＋節目で上位Tierを習得**（＝滑らかな成長で無駄なし＋"ここまで行けば上位魔法"という到達目標＝モチベ設計）。例）キュアー 10 で「キュアー1」→ 20 で「キュアー2」→ …。
- **投資量＝威力＋到達Tier＋MP寄与**。深く振るほど強い呪文まで届く。
- **1プール制**：基本5ステ（体力/魔力/力/素早さ/精神）と**同じポイントを奪い合う**（回復に振る＝攻撃・精神が下がる＝役割分化の担保）。
- リスペック（ゴールド消費）で振り直し可。
- **呪文リスト（★＝MVP実装／☆＝将来拡張。全ライン習得値10始まり・Tier制。数値は仮）**：

  | ライン | 効果 | 区分 |
  |---|---|---|
  | ファイアー (Fire) | 攻撃魔法・単体（精神で軽減／一部%最大HP） | ★MVP |
  | キュアー (Cure) | HP回復 | ★MVP |
  | スリープ (Sleep) | 敵を眠らせ行動阻害（命中/耐性/持続あり） | ★MVP |
  | ストレングス (Strength) | 力アップ（自己強化バフ。硬い敵の突破に有効） | ★MVP |
  | 状態異常（毒 等） | 毒/麻痺などの継続ダメージ・行動阻害（属性なしでも成立） | ☆将来（優先度高） |
  | ディフェンス（被ダメ軽減）/ スピード | 守り・加速の支援バフ | ☆団体戦フェーズ |
  | AoE / 属性 / デバフ | 全体攻撃・炎氷雷・敵弱体 | ☆横成長 |

  - 攻撃魔法は**属性なし＝単体1本のみ**（少数でlaunch・9章）。名称は英語ベース。Tierは累積投資で上昇（ファイアー1=10, 2=20…）。1プール制ゆえ「深いTier＝他が薄くなる」トレードオフ。

### 3.6 装備（ゴールド & ドロップ）
- 装備 = **「+いくつ強い」ではなく「ビルドの型」**（剣=力型、杖=魔法型、一長一短）。**横の軸**。
- **防具（鎧・盾）＝物理防御の担い手**（守備ステ廃止に伴い、物理ダメージ軽減は装備で行う＝3.5.1）。重装＝物理に強いが別のコスト、軽装＝別の型、という横の型。魔法ダメージ防御は魔力ステ・状態異常耐性は精神ステが担うが、将来"魔法耐性の装備（ローブ/護符）"も型として追加余地。
- 入手は2系統＋保険：
  - **ドロップ**（修行/ミニゲームのクリア報酬。サプライズ・型集めの楽しみ）
  - **店/クラフト**（ゴールドで狙い撃ち。運に人質に取られない）
  - **ピティ（天井）**（運が悪くても続ければいつか入手 → 全員到達可能）
- 装備は横の軸なので、**RNGでも「運=強さ」にならない**（弱いんじゃなく型が少ないだけ）。
- 新装備は「強さ」でなく「**新しい型**」を増やす（=横の拡張 → インフレ・年功ロックを回避）。

### 3.7 キャラクター画像／アバター（見た目・個性）
- 観戦型で個性の出し所が少ないため、**見た目は数少ない自己表現の場**。ここは全員に開放する。
- **2系統**：
  - **プリセット**：こちらが用意した数枚から選択。**アプリ同梱アセット**として持つ（egress = 0・恒久無料・ネット不要で即表示）。デフォルトはここから選ぶ。
  - **ユーザーアップロード**：**全員無料で開放**（自分でイラストを描く/用意して独自性を出せる）。課金枠にはしない。
- **管理＝Supabase Storage**。無料を守る実装ルール：
  1. アップロード時に **256×256程度のwebpへ強制圧縮**（生画像は配信しない。クライアント側で圧縮してから上げる）。1枚 ≈ 15〜30KB。
  2. サイズ/解像度の**上限をクライアント＆Storageポリシー両方で強制**。
  3. **1キャラ1枚・差し替えは上書き**（ストレージ肥大防止）。
  4. アプリ側**ローカルキャッシュ**（`cached_network_image`）＋StorageのCDNキャッシュ。アバターは静的（滅多に変わらない）ので、egressは「ユニークな(端末, アバター)」しか食わない。
- **コスト方針**：少人数のうちは**無料枠で回る**（保存量は問題にならず、egressもキャッシュで大幅圧縮）。ユーザーが増えたら **Supabase Pro → 従量へツマミを上げるだけ**（構成不変・13.7と同じ）。
- **モデレーション**（不適切画像対策）はコストと別問題として残る。観戦で他人に見えるため最低限：
  - まず「**通報 → 差し替え/BAN**」の事後対応のみ（軽い・無料）。
  - NSFW自動判定は荒れ始めてから後付け（従量課金なので初期は入れない）。
- **データモデル**：`avatar_type ('preset' | 'custom')` + `avatar_ref`（preset: `'preset_03'` / custom: storage path）の1組で両対応。アップロード対応を後付けしても既存データを壊さない。

## 4. バトル

- **RPG方式・サーバー自動計算**。プレイヤーは結果ログを後から観る。
- 演出：**クラシック（レトロ）RPG風**（黒背景・テキスト主体・最小限の演出・被弾で画面揺れ）。通信も軽い。※特定作品の固有名/画像/音楽/モンスター等は使わず、HP等の**汎用RPG慣習のみ**採用（IP対策）。
- **命名ポリシー（IP対策・確定）**：呪文/攻撃名は**英語・元素ベースの一般名を標準採用**（Fire／Flame／Thunder／Blizzard／Ice／Heal 等）。理由＝一瞬でイメージでき、著作権も安全。オリジナル造語は覚えてもらいにくいので避ける。既存作品の**造語**（メラ・ギラ・ホイミ／ファイガ・ケアル等）は当然NG。※「ファイア」は一般名でOKだが「ファイガ」等の固有変化形はNG。
- **相性ベース**（後述のバランス方針）。

### 4.1 戦闘の最小キット（MVP・特技なし）
- **行動タイミング＝ATB（持ち点式ターン順 / CTB）で確定**。素早さで持ち点（ゲージ）が溜まり、閾値を超えた者が行動→消費…の繰り返し＝**素早さが高いほど手数が増える**（連続行動あり＝観戦のスリル）。**完全に決定論**（同シードなら同結果＝運に見えない）。リアルタイムATBの重さ（時間tick・キャスト時間管理）は避け、**持続（状態異常など）は行動回数/ターンで数える軽量版**。詳細は4.1.1、運・分散の方針は4.1.2。**サーバー内の決定論シミュレーション**で実装（観戦・バッチ・安さは不変／リアルタイム操作ではない）。
- 各キャラは**装備武器に応じた基本攻撃**を自動で行う（特技・スキルは無し）：
  - **物理攻撃**（剣・斧など）：ダメージ＝力ベース、**鎧（装備）で軽減・素早さで回避され得る**、MP不要。ビルドが物理でなくても使える基本行動（＝MP切れ時のフォールバック）。
  - **魔法攻撃**（ファイアー等）：ダメージ＝攻撃魔法ライン投資ベース、**守備を無視するが精神で軽減**（＋最大HP割合成分）、**MP消費・回避不可（精神で耐性）**、**MP切れで失速**（弱い物理に戻る）＝魔法型の弱点。
- **物理攻撃タイプは武器で表現**（3.6「剣＝力型／杖＝魔法型」）。魔法は習得した呪文をエンジンが状況判断で発動。
- 特技（非魔法の技）は **launch では入れず**、将来の横成長(6章)の弾として温存（呪文はMVPからあり）。

### 4.1.1 行動タイミング：ATB＝持ち点式ターン順（CTB・確定）
- **狙い**：素早さで**手数が変わる**（＝観戦がスリリング＆素早さが死にステにならない）を、**完全決定論**（運に見えない／再現・検証可）かつ**軽量実装**（リアルタイムATBの時間tick・キャスト時間管理を避ける）で実現する。
- **擬似コード（仮）**：

```
# 決定論CTB（持ち点式ターン順）
rng = seededRNG(seed)            # 試合ごとに新規シード→毎試合違う結果／シードは記録＝再現・検証・再生可
combatants = [A, B]              # ★最初から「リスト対リスト」で書く（団体戦3v3・ダンジョンのグループ戦へ無改修で拡張）
for c in combatants: c.gauge = 0

loop until 決着 or ターン上限:      # 上限＝PvP膠着防止（3.5.1/4.2）
    for c in 生存(combatants):
        c.gauge += speedToGauge(c.素早さ)   # ★逓減込み＝2倍速≠2倍手数（速度一強を防ぐ）
    ready = { c | c.gauge >= THRESHOLD }
    if ready 空: continue
    actor = ready の中で gauge 最大         # 同値は 素早さ→固定ID の順で決定論タイブレーク
    actor.gauge -= THRESHOLD
    行動 = engineが定石で自動選択           # 例: HP50%未満→キュアー／開幕→バフ／強敵→スリープ／通常→ファイアー（4.2・12章）
    resolve(行動, rng)                     # ダメージ±10〜15%等の"小さな乱数"はここで rng から引く
    log.push(event)                        # eventLog＝クライアント再生の契約（13.1）
return { winner, seed, eventLog }
```

- **速度一強を防ぐ逓減**：`speedToGauge` は線形でなく逓減（例：平方根的）＝**2倍速≠2倍手数**。加えて①**魔法は必ず当たる**（回避は物理のみ＝素早さは対魔法に無力＝弱点を残す）、②**1プールで速度型は紙**（速度に振る＝力/体力/魔力が薄い）。→ 素早さは強いが天敵を持つ。
- **拡張性**：`combatants` を**リスト対リスト**で書くことで、団体戦（8章 3v3）・ダンジョンのグループ戦（12章）を**エンジン改修なし**で載せられる（1v1はN=1の特殊ケース）。

### 4.1.2 運・分散の方針（確定）
- **決定論と運は両立する**：`battle()` の決定論（13.4）は「同シード→同結果」の意味。**シードを試合ごとに新規発行**すれば**同カード再戦でも結果は揺れる**（＝サッカー/野球の手触り）。発行済みシードは**記録**するので再現・検証・観戦再生が可能。＝**運を消す仕組みでなく、運を記録・再現する仕組み**。
- **良い運＝小さな乱数を多く／悪い運＝一発の大博打を避ける**：
  - ダメージ乱数 **±10〜15%** を全攻撃に薄く乗せる（主力の分散源）。大数の法則で**強いビルドが期待値で勝つが、たまに揺れる**。
  - クリティカルは**低確率・中倍率**（1.5倍程度。2倍以上にして一撃で試合を決めない）。
  - 物理回避は**控えめ**（魔法は当たる）。
- **スリープ＝最大の分散源＝アップセット装置**（3.5/9章）：格上を食う"パンチャーズチャンス"。ただし**精神で自己修正メタ**が働くので運ゲーにならず読み合いになる。
- **フォーマットで運を出し分け**（5.3）：予選リーグ（複数試合）＝**運を平均化**（実力が出る）／トーナメント（一発勝負）＝**運を増幅**（ドラマ）。＝リーグ表は公平・カップ戦は何が起きるか分からない、という現実の構造。
- **勝率ターゲット（仮・実測調整の指針）**：同格（互角ビルド）**45〜55%**／1ティア格上**65〜70%**／明確な格上**80%前後**。**100%は作らない**（「強い人が基本勝つ」を維持しつつ希望を残す＝3.5/7章）。
- **測って決める**：観戦型＝**全試合のビルドと勝敗の完全データ**があるので、運の量（アップセット率）を**実測→毎シーズン調整**（9章 live-ops）。勘で数字を決めない。
- **番狂わせは理由が見える**：アップセットは「スリープが入った」「クリティカルが刺さった」等**ログで説明**できること。理由が見えれば運は**ドラマ**に、見えなければ**イカサマ**に見える（＝運を入れても因果は必ず見せる）。

### 4.2 魔法のロールと戦闘での働き
- **攻撃魔法（ファイアー）**：魔法の攻撃極。**威力＝Tier基本＋術者魔力−受け手魔力**（＝**受け手の魔力で軽減**・精神では防げない／鎧は無視／下限あり／%最大HP成分）。属性なし・単体1本。MVP実装。
- **回復魔法：MVPで実装**。理由＝**ソロ（派遣ダンジョン）で明確な価値**：戦いながら自己回復 → 自然回復に頼らず**MPが続く限り連続で潜れる＝uptime延長＝取り分増**。「攻撃全振り＝バーストだが回復薬/自然回復頼み」vs「回復持ち＝自給自足で長く潜る」という**バースト対サステインのビルド分化**を生む。
- **補助魔法**：MVPは**ストレングス（力アップ）のみ**＝硬い敵（重装甲で物理が通りにくい相手）の突破に有効で、ソロで元が取れる唯一のバフ。**ディフェンス（被ダメ軽減・※守備ステ廃止ゆえ"軽減%付与"）／スピードは団体戦フェーズ**（守り・加速は味方に掛けてこそ／8.3）。
- **状態異常（スリープ）＝アップセット/ジャイアントキリングの装置**：効けば**格上にも勝つ可能性が出る**＝underdogのパンチャーズチャンス（上限なし課金モデルに希望を足す／期待値は強者有利のまま＝「強い人が基本勝つ」は崩れない）。
  - **健全化の鍵＝精神**：効くのは"攻撃全振りで精神を捨てた一本槍"、**精神を積んだバランス型には効きにくい** → 「精神を軽視した強さ」を罰する（＝精神の存在意義を強化・glass cannon ⇄ 精神型の読み合い）。
  - **"lock"でなく"chance"に留める**：成功率＝**魔力(攻) vs 精神(防)の相対**（floor/ceiling付き・0/100%にしない）／持続限定・被弾で起きる／再睡眠に逓減・耐性上昇（perma-lock防止、特にPvP）。
  - **スリープゲー化を防ぐ＝自己修正メタ**：カーブは**魔力≈精神の周辺で急**（魔力<精神＝ほぼ効かない[低floor・0%ではない]／魔力>精神＝高確率）。→ スリープが流行れば皆が精神を上げ、スリープが死に、また廃れる、という**循環メタ**で一強化しない。しかも**精神は魔法ダメージ軽減も兼ねる**ので「対スリープ税」でなく正当な防御投資。1プールの機会費用（攻撃側＝魔力/眠り線、防御側＝精神）が両者にかかりタダにならない。**毎シーズン、スリープ絡みの勝率を監視して曲線を調整**（9章 live-ops）。
  - **PvE＝cheese歓迎**（格上ボス攻略の鍵）／**PvP＝理不尽にならない範囲で慎重に**（睡眠耐性持ちの敵も variety としてアリ）。
  - **毒・麻痺等は将来拡張**（優先度高・属性なしでも成立）。
- **balanceガード**：MPは有限（回復も枯れる）＋同投資量ならアタッカーが火力で上回る → **純回復は無敵にならない**。PvPトーナメントの膠着はターン/MP上限で防ぐ。
- **自動戦闘の呪文運用（MVP＝エンジン自動判断）**：観戦型でプレイヤーは戦闘中に介入しない → **エンジンが定石で自動判断**して唱える（例：`HP50%未満→キュアー／開幕→バフ／強敵→スリープ／通常→ファイアー`）。※将来、プレイヤーが優先度を事前設定する「作戦」システムを**第3の戦略入力**として足す余地は残す。
- 習得は「呪文ライン別の修行式」（3.5.2）。攻撃寄り＝魔法使い型／回復寄り＝僧侶型に自然分化。

## 5. 大会構造

### 5.1 2軸
- **個人戦**（マイペース層）と **団体戦**（社交層）の2軸。両方楽しめる。
- **段階リリース：まず個人戦だけ → ユーザーが増えたら団体戦（=コミュニケーション機能）を追加**。

### 5.2 ディビジョン（昇格・降格ラダー）
- サッカーのJ1/J2方式。**下位リーグで勝ち上がる → 上位リーグへ挑戦**。
- 効能：
  - **脱落対策**：全員に手の届く目標が常にある（「まず昇格」）。
  - **マッチングが自動**：ディビジョン＝対戦帯。強さ順に振り分く（課金/廃人が速く上がっても下位帯を蹂躙しない）。

- 階層を登り切った先に最高峰の大会（W杯相当）。

## 6. インフレ対策・新規体験（設計の背骨）

- **基本方針：時間 or お金 → 強さ（上限なし・累積投資が強さ）**。公平性の均等化は狙わない（世の多くのF2Pが証明済みの王道）。
- ただし「**新規がゲームにならない**」「**桁が暴走する**」の2つは別問題として対処する。

| 問題 | 対策 |
|---|---|
| 数値インフレ（桁の暴走） | ソフトキャップ曲線（上位ほど伸び鈍化）/ 装備は横拡張（型を増やすだけ） |
| 新規が古参にワンパンされる | **ディビジョン**（強さ帯で自動マッチング）＝新規は新規と、廃課金は廃課金と当たる |
| 新規・無課金の「勝てる目標」 | **シーズンリセット＋ディビジョン**＝自分の帯なら誰でも優勝を狙える |
| 桁が寝てからの成長実感 | 縦が鈍ったら**横成長**（新スキル/型/称号/コレクション）で供給 |

- **「トップ」の再定義**：全体の頂点は**時間・お金の累積が最大の者**。無課金/新規の到達目標は**「自分のディビジョンでの優勝」**。ディビジョン＋シーズンがこれを毎月供給する。

## 7. 課金方針

- **基本：時間を買う（pay-to-skip）を主軸としつつ、強さに効く課金（性能）も許容する。**「時間 or お金 → 強さ」の方針に沿う。
  - **回復薬（体力回復）**：課金で購入可。**多く派遣を回す＝速く強くなる**に効く（＝時間を買うの延長）。
  - その他OK：経験値/時短ブースト、育成枠・キャラ/チーム枠の追加、シーズンパス、**見た目（武器グラフィック・称号・名前の色・レトロスキン）**、動画広告（時短 & 収益）。
- **新規体験の担保**：強さの差そのものは許容し、**ディビジョン＋シーズンリセット**で吸収する（廃課金は上位帯へ、新規は下位帯で完結・自帯で優勝可能）。
- **広告**：忙しい人の間口＋収益を兼ねる。
- **ゴールド／回復薬**：売ってよい。

## 8. 団体戦（将来実装）

> **位置づけ＝エンドゲーム／最上位の遊び。** 個人戦は「時間・お金 → 強くなる」power fantasy（序盤〜中盤の主動機・間口）。ソフトキャップで**強さの伸びが鈍った最上位層**に、**戦略・編成・コミュニケーション**という別軸の面白さを供給するのが団体戦。→ 「上位ほど成長が寝て飽きる」というソフトキャップの弱点の**受け皿**になり、10章の段階リリース（個人戦先行 → 団体戦後付け）に**必然性**を与える（団体戦をリリースする頃には、それを必要とする最上位層が育っている）。

### 8.1 2層構造
- **クラブチーム**：1ユーザーが持つ単位。※MVPは「1ユーザー=1分身」だが、**団体戦フェーズで1ユーザーが複数キャラを育成可能に**（僧侶/魔法使い/戦士を自前で編成できる＝ロール多様性を自給自足・飽き防止・キャラ枠課金と整合[7章]）。合同チームは複数プレイヤーの集まり。
- **合同チーム**：発起人が**他プレイヤーのキャラをスカウトして編成**するオールスター。複数プレイヤーの協力。**W杯の頂点**。
- → トロフィー2系統（クラブ優勝 / 合同チーム優勝）。ログイン動機が二重に。

### 8.2 メカニクス案
- **3 vs 3** を想定、10人規模でチームを組み**ローテーション**。相手の編成を読んで対策編成。
- **ダメージ持ち越し（消耗）**：次戦までに全回復しない → 「誰をいつ出すか」の**継投マネジメント**が戦略に。
- **チャットで相談**しながら出場順を決める（非同期 × 社交）。操作権は **C案（ビルドは各自・当日の出場順や作戦はチャット合議でキャプテンが確定）** を想定。
- **スカウトを“市場”化**：ランキング/移籍市場から強キャラを物色 → 勧誘 → 受否。**信用が貯まる**。人間関係そのものがコンテンツ。
- **軽い拘束ルール**：大会エントリー後はメンバー固定／無断離脱は信用ダウン（荒れ防止）。

### 8.2.1 隊列（ポジション）システム
- 団体戦では**並び順（前衛〜後衛）**を持つ。**前のキャラほど物理攻撃を受けやすい**（物理アグロが前方に集中）。**魔法攻撃は隊列に関係なく当たる**（位置で守れない）。
- 帰結＝**役割と隊列が噛み合う**：
  - **前衛**：**重装甲（鎧）＋HP**を盛る物理の壁。後衛を物理から守る。
  - **後衛**（回復役/魔法アタッカー等）：守備は低くてOKだが、**精神は必須**（魔法が位置を貫通するため）。
- → 3.5.1で作った**守備／精神の使い分けが隊列戦略として意味を持つ**。個人戦の「物理/魔法・守備/精神の読み合い」が、団体戦では**「隊列をどう組むか」にスケールアップ**：
  - 物理偏重の敵 → 前衛の壁で受け切る。
  - 魔法偏重の敵 → 隊列を貫通して後衛の紙耐久を狙える → **後衛の精神** or 敵魔法役を早く落とす対抗編成が要る。
- **物理アグロ配分**：100%前ロックでなく**前方重み付け**（例：前70%/中20%/後10%、前衛が倒れたら繰り上がり）でソフトに。
- 隊列（誰を前後に置くか）は**キャプテンが決める事前の戦略入力**（8.2のC案「出場順/作戦をキャプテンが確定」の一部）。
- ※ソロ（個人戦）には隊列なし（個人戦は単純なまま。隊列の駆け引きは団体戦の深み＝二層設計と一致）。

### 8.3 多様性の強制
- **戦士だけのチームは勝てない**設計。色々なタイプが要る。
  1. 相性網が mono-comp を罰する（偏り＝穴、多様＝堅い）
  2. 役割の相互依存（回復/壁/火力/支援が噛み合って初めて機能）
  3. 同タイプ重ねに逓減 / 多様性ボーナス
- 副作用：**「回復型が足りない、スカウトしよう」**＝チーム編成とスカウトの動機を直接生む。

## 9. バランス方針（最大の難所）

- **どれが強い/弱いではなく「相性」**。「このタイプ vs このタイプはこっちが勝ちやすい」。
- **相性はラベルでなく戦闘式から創発**：各ステに「攻めの役割」と「弱点」を両方持たせる。
  - 例：素早さ→行動順/回避（先手・躱し）、力→物理バースト（鎧を割る）、体力→HP（粘り）、魔力→魔法ダメージ＆状態異常（低魔力を溶かす／精神薄を眠らせる）、精神→状態異常耐性、鎧→物理軽減。
  - **創発する三すくみ（数字から自動で出る）**：魔法使い(高魔力・紙) ＞ 重装タンク(低魔力を溶かす) ＞ 速攻物理(鎧で受け粘る) ＞ 魔法使い(先手バーストで紙を潰す) → 1周。**「魔法＝強い」ではなく、1プールで何かに振れば何かが薄くなり、必ず天敵がいる**（例：魔力に振った魔法使いは素早さ・体力が薄く、速攻物理に先手で沈む）。
- **完璧な事前バランスは諦める**。代わりに **観戦型＝全試合のビルドと勝敗の完全データ** を武器に、**毎シーズン勝率をデータで均す（生きたメタ）**。
- **ソフトな相性**（100:0でなく60:40）。相手を選べないので「天敵に当たって即詰み」を避ける。
- **少数タイプで launch → データで均してから拡張**（最初から多タイプは破綻する）。
- ※具体数値は本質的に**launch後の継続調整（live-ops）**。企画書では「相性ベース/データで毎シーズン調整」という**方針**を確定。

## 10. 段階的スコープ（実装ロードマップ案）

### MVP（最初に出す）
- 1ユーザー = 1キャラ、自由ステ振り（物理3ステ＋魔法ライン・1プール）、衰えなし（persistent）。
- 育成：主軸＝**派遣ダンジョン**（体力ループ＋自然回復＋回復薬）＋広告/ミニゲーム。ゴールド＋装備（店/ドロップ）。ソフトキャップ曲線（上限なし）。
- 魔法：**ファイアー（攻撃・単体）＋キュアー（回復）＋スリープ（眠り）＋ストレングス（力アップ）**の4ライン（修行式習得）。属性・状態異常(毒等)・他バフは将来。
- **個人戦のみ**：予選リーグ → トーナメント、ディビジョン（1〜2階層）、シーズン制、練習試合。
- バトル：自動計算 + テキストログ（クラシックRPG風）。少数タイプ。
- 課金：時短/枠/見た目/広告/回復薬。

### 拡張（ユーザー増加後）
- 団体戦（クラブ/合同チーム）、スカウト/移籍市場、チャット、消耗システム。
- **補助魔法・ロール深化**（回復魔法はMVPで導入済）、**1ユーザー複数キャラ育成**（キャラ枠課金）。

## 11. 確定した重要決定（サマリ）

- キャラ：**1ユーザー=1分身、自由ステ振り、衰え（加齢）なし**。
- 見た目：**プリセット（アプリ同梱）＋ユーザーアップロード（全員無料）**。Supabase Storageで管理・少人数は無料枠、増えたら従量。課金枠にしない。
- 強さ：**時間 or お金 → 強さ（日次上限なし・累積投資が強さ）**。伸びは**ソフトキャップ曲線**で鈍化（桁の暴走抑制が目的、公平化ではない）。バトルはビルドの腕で差。
- インフレ／新規体験：**ソフトキャップで桁を抑制**。**ディビジョン＋シーズンリセット**で新規体験を担保（自分の帯で優勝可能）。
- 課金：**時間を買う＋性能課金も許容（回復薬等）**。強さ差はディビジョン＋シーズンで吸収。
- 育成：**複数チャネル（プラグイン式）。主軸は派遣ダンジョン（体力ループ＋自然回復＋回復薬）**。日次上限なし・投資量で伸びる。
- 経済：**XP=縦（ステ）/ ゴールド=横（装備=型）**。装備は全員到達可能・有界・ドロップ+店+ピティ。
- ステータス：**基本5ステ（体力・魔力・力・素早さ・精神／HP=体力×n・MP=魔力×n。魔力は魔法威力/状態異常成功率も担う＝力↔魔力の対称）＋魔法ライン（攻撃/回復/状態異常/バフ）を1プールから配分。物理防御＝鎧（装備・横）／**魔法ダメージ防御＝魔力（魔力vs魔力）／状態異常耐性＝精神（魔力vs精神）**＝単一ステで魔法無敵にならない。魔法は修行式（投資で威力↑＋閾値でTier習得）。非魔法の特技なし**。
- 大会：**個人戦先行 → 団体戦後付け。ディビジョン昇降格 + シーズン**。
- バトル進行：**ATB＝持ち点式ターン順（CTB）。素早さで手数が変わる・完全決定論・軽量実装（持続は行動回数で数える）。エンジンはリスト対リストで書き団体戦/グループ戦へ無改修拡張**。
- 運・分散：**運は毎試合の新シードで供給（記録＝再現可）。小さな乱数を多く＋スリープ＝アップセット装置＋フォーマットで平均化↔増幅。勝率は同格五分〜明確格上8割で100%は作らない。観戦型の完全データで実測調整**。
- 遊びの二層：**個人戦＝時間/金で強くなる power fantasy（序盤〜中盤の間口）／団体戦＝最上位の戦略・社交エンドゲーム（伸びが鈍った層の受け皿）**。
- バランス：**相性ベース。観戦型の完全データで毎シーズン調整。少数で launch**。

## 13. 技術構成

### 13.1 大前提：サーバー権威（server-authoritative）
- **バトルのロジックはアプリではなくサーバーで動かす。** 理由が企画に直結：
  - 非同期observe：バトルは指定時刻に自動進行。**プレイヤーがオフラインでもサーバーが計算**して結果を残す。
  - **不正防止**：クライアント計算は改ざんし放題 → 競争ゲームは成立しない。
  - NPC/ゴースト/トーナメントも全部サーバーのスケジュール処理。
- → **アプリは「結果を観るビューア + 育成/ビルドの入力」**。バトル計算はサーバーが**1回だけ**実装。
- **アプリ＝戦闘イベントログの再生役**。サーバーが「誰が何して何ダメージ」のログを返し、アプリはそれをアニメ再生するだけ。
  → アプリとサーバーの言語が違ってよく、技術選定が独立する。

### 13.2 クライアント（アプリ）：Flutter
- **iOS/Android を1コードベースで**。ネイティブ別々（Swift+Kotlin）は避ける。
- **Flutter 本命**。理由：本作は「メニュー/ブラケット/ステ振り/ショップ/ログ表示が9割 + シンプルな戦闘再生が1割」のアプリ寄り。ドラクエ3風の最小演出に十分。
- **ミニゲームも同じFlutterで**：
  - オセロ/ナンプレ/スライドパズル/カード等 → 素のFlutter（`CustomPainter` + `GestureDetector`）
  - ゲームループが要るもの（マッチ3・落ち物） → **Flame**（Flutter製2Dゲームエンジン）
- **「全部1スタック（Dart/Flutter）で書ける」のが最大の利点**。Unityを混ぜるとハイブリッド化して重く・面倒 → 混ぜない。

### 13.3 バックエンド：Supabase
- 本作は**リレーショナル**（players / characters / equipment / tournaments / matches / divisions / 順位表）→ **Postgres（Supabase）がFirestore（NoSQL）より素直**。
- Supabaseが最初から持つもの：**Postgres / Auth / Realtime（チャット用に後で効く）/ Storage / Edge Functions（TS）/ pg_cron / プッシュ連携**。
- **プッシュ通知は FCM（無料・スケールしても無料）**。
- 限界：重い計算（何万試合の一括シミュ）はEdge Functionの時間制限がきつくなりうる。**MVPは問題なし**。スケールしたら戦闘エンジンだけ外部ワーカーに逃がす。

### 13.4 戦闘エンジン
- **純粋関数**：`battle(buildA, buildB, seed) → { winner, eventLog }`。グラフィックなしの計算のみ → 軽い・テスト容易。**1回だけ実装**。
- **決定論（determinism）**：同じ (ビルドA, ビルドB, シード) なら必ず同じ結果。**シード固定のRNG**を使用。入力とシードを保存すれば再現・検証可、再生アニメと結果も一致。
- **ステートレスに保つ** → 将来、重くなったらこの関数だけ安いコンピュート（Fly.io / Cloud Run / Oracle無料VM等）へ差し替え可能。

### 13.5 毎日の対戦（バッチ処理）
```
① プレイヤーが事前にビルド/編成を登録（DB保存）
② 定時(例:毎日21:00)に cron(pg_cron) が発火
③ バッチ:
    - その日の対戦カードを決定（各ディビジョンのリーグ/トーナメント表）
    - 各カードで battle() を実行（決定論シミュ）
    - 結果 + 戦闘イベントログをDB保存
    - 勝者を次ラウンドへ / 順位・昇降格を更新
    - プッシュ通知「結果が見れるよ」
④ プレイヤーは好きな時間にアプリを開く → ログ取得 → アニメ再生
```
- **スケジューラ**：MVPは Supabase の **pg_cron → Edge Function(TS)**。プロト段階は GitHub Actions の cron でも可。
- **冪等性**：各カードに「処理済みフラグ」、トランザクションで囲む（再実行で二重処理しない）。
- **スケール**：全カードをループで回すだけ。数式ループなので何万試合でも速い。増えたらカードをキューに積んでワーカー並列。**MVPは単純ループで十分**。
- トーナメントは前ラウンド結果に依存するので、**遅延計算でなく定時バッチで一斉確定**が筋が良い（順位表・ブラケットの整合）。

### 13.6 不正対策
- バトルはサーバー計算（改ざん不可）。
- **ミニゲームの報酬（経験値/ゴールド）はサーバー検証 or 上限つき**。「報酬は参加ベース・日次上限」設計なので、上限で頭打ちにすれば不正の旨味も消える。

### 13.7 コスト戦略：無料スタート → 増えたら課金（作り直し不要）
- **構成を変えずにツマミ（プラン）を上げるだけ**でスケールする。
```
段階0(MVP/少人数): Supabase無料枠 + FCM        … ほぼ$0（ストア代のみ）
段階1(増加):       Supabase Pro $25/月          … プランを上げるだけ
段階2(さらに増加): 戦闘エンジンだけ別ワーカーへ … 既に純粋関数で分離済＝差し替え容易
段階3(大規模):     従量課金
```
- **避けられない実費（ローンチ時）**：Apple Developer **年$99**（iOS必須・主）/ Google Play **初回$25**。開発・テストは無料、公開時のみ。
- **無料枠の注意**：無料Supabaseは約1週間無アクセスで一時停止（ローンチ後は毎日バッチで止まらない）。課金は主に egress（帯域）で発生。
- **設計が安い理由**：**非同期observe設計はリアルタイム対戦サーバー不要**。「定時バッチ + DB読み取り」だけ → まったり観戦型（魅力）がそのまま running cost の安さに直結。
- **安いまま伸ばす習慣**：戦闘エンジンはステートレス独立関数／egress節約（画像キャッシュ・無駄なAPIを叩かない・ログは必要分のみ）／重い処理はバッチでまとめる。

### 13.8 MVPスタック（確定）
> **Flutter（+ Flame：本体・ミニゲーム・戦闘再生を全部）+ Supabase（Postgres / Auth / Realtime / Edge Functions / pg_cron）+ FCM プッシュ + 定時バッチで決定論バトルエンジン（純粋関数）。**
