## 0. 全体戦略（なぜこの順番か）

**エンジンファースト × インフラ前に検証。**

- 戦闘エンジンは**純粋関数**（`battle(A, B, seed) → {winner, eventLog}`）＝**インフラ0で作れる**。しかも**最大の難所＝バランス**（企画書9章）。
- よって **UI・DB・大会を積む前に、エンジンとバランス検証（M0〜M2）を先に終える**。核が正しいと分かってから周辺を乗せる。
- もし三すくみが崩れる／最強ステ・死にステが出る等の問題があれば、**UIを1画面も作る前に安く直せる**。インフラを積んだ後に気づくと高い。
- 決定論（企画書13.4）ゆえ、M2の総当たりシミュは何千試合でも一瞬＝**launch前に9章のlive-ops調整を予行できる**のが最大の武器。

**基本原則**：各マイルストーンは「**観測できる成果物**（動く／テストが通る）」で締める。

### M0 — 契約を固める（設計・実装前）
エンジン／DB／Flutterの**共通契約**を先に確定する。CTBが決まった今、ほぼ機械的に書ける。

**成果物**
1. **ビルド入力スキーマ**（`battle()` の入力＝1キャラのビルド。DB保存形・クライアント入力形と同一）
2. **eventLogスキーマ**（`battle()` の出力＝再生の契約）
3. **戦闘定数config**（倍率・±%・THRESHOLD・逓減カーブ。**全部仮値でよい**。後でM2/9章で実測調整）

**ビルド入力スキーマ（ドラフト・仮）**
```jsonc
{
  "characterId": "uuid",
  "level": 12,
  "stats": {            // 基本5ステ（1プールから配分。同スケール／企画書3.5.1）
    "vit": 10,          // 体力  → maxHP = vit × 10（仮）
    "mag": 8,           // 魔力  → maxMP = mag × 5（仮）＋魔法威力/軽減/状態異常成功率
    "pow": 6,           // 力    → 物理攻撃
    "spd": 9,           // 素早さ→ CTB持ち点の溜まり＋物理回避
    "men": 5            // 精神  → 状態異常耐性（のみ）
  },
  "spellLines": {       // 魔法ライン修行値（同じ1プールから／企画書3.5.2）
    "fire": 20,         // Tier = floor(value/10) 等（仮）
    "cure": 10,
    "sleep": 0,
    "strength": 0
  },
  "equipment": {        // ゴールド軸＝横の型（企画書3.6）
    "weapon": "sword_iron",   // 力型/魔法型など「型」
    "armor":  "mail_leather", // 物理防御の担い手
    "shield": null
  }
}
```

**eventLogスキーマ（ドラフト・仮）**
```jsonc
// battle() が返すイベント列。Flutterはこれを順に再生するだけ（企画書13.1）。
[
  { "t": 0, "type": "battle_start", "combatants": ["A","B"] },
  { "t": 3, "type": "gauge_ready",  "actor": "A" },
  { "t": 3, "type": "attack",  "actor": "A", "target": "B", "weapon": "sword_iron",
    "damage": 42, "crit": false, "hpAfter": 118 },
  { "t": 5, "type": "cast",    "actor": "B", "spell": "fire", "tier": 2, "mpAfter": 12 },
  { "t": 5, "type": "damage",  "actor": "B", "target": "A", "amount": 55, "kind": "magic", "hpAfter": 90 },
  { "t": 7, "type": "cast",    "actor": "A", "spell": "sleep", "tier": 1 },
  { "t": 7, "type": "status_apply", "target": "B", "status": "sleep", "duration": 2, "success": true },
  { "t": 9, "type": "miss",    "actor": "B", "target": "A", "reason": "evade" }, // 物理のみ
  { "t": 12,"type": "cast",    "actor": "A", "spell": "cure", "heal": 40, "hpAfter": 130 },
  { "t": 20,"type": "ko",      "target": "B" },
  { "t": 20,"type": "battle_end", "winner": "A", "seed": 123456 }
]
```
> `t` はCTBの内部時刻/行動カウンタ（企画書4.1.1）。イベント種別は実装で確定。

**完了条件（DoD）** ✅ 完了
- [x] 3スキーマをファイル化（`engine/src/schema.ts`＝型・`formulas.ts`＝仮値config＋装備カタログ）
- [x] 「5ステ＋4ライン＋装備3枠」で1ビルドが過不足なく表現できる（`CharacterBuild`）
- [x] eventLogの全種別が「攻撃/魔法/回復/状態異常/回避/撃破/開始終了」を再生に必要な最小情報で覆う（12種の discriminated union）

### M1 — 戦闘エンジン（純粋関数・インフラ0）★最優先
**実装言語＝TypeScript**（Edge Function前提・企画書13.9／将来ワーカー移設も見据える）。グラフィック無し・計算のみ。

**実装内容**
- **CTB持ち点ループ**（企画書4.1.1の擬似コード）：`combatants` を**リスト対リスト**で書く（団体戦へ無改修拡張）。1v1はN=1。
- **シード固定RNG**（決定論。同シード→同結果）。試合ごとに新規シードを外から受け取る。
- **物理ダメージ式**（力×武器・鎧で軽減・素早さで回避・±10〜15%乱数）
- **魔法ダメージ式**（`Fire基本＋術者魔力−受け手魔力`・下限あり・%最大HP成分・回避不可）
- **4魔法ライン**：Fire（攻撃）／Cure（回復）／Sleep（状態異常）／Strength（バフ）
- **状態異常（sleep）**：成功率＝魔力(攻) vs 精神(防) の相対（floor/ceiling・0/100%なし）・持続限定・被弾で起きる・再睡眠に逓減
- **自動判断AI（定石）**：`HP50%未満→Cure／開幕→Strength／強敵→Sleep／通常→Fire`（企画書4.2・仮ロジック）
- **ターン/MP上限**（PvP膠着防止）

**完了条件（DoD）** ✅ 完了（`engine/src/battle.ts`・`rng.ts`／テスト6件 green）
- [x] `battle(input, seed)` がCLIで走り、eventLogを出力（`deno eval` で確認済）
- [x] **決定論テスト**：同 (input, seed) が必ず同結果・同ログ（JSON完全一致で検証）
- [x] 境界テスト：低魔力が魔法に溶ける／sleepは低精神に効き高精神に効きにくい（魔力≈精神で50%）／非魔法型はcast0・attackのみ／大半の試合で決着
- [x] `gaugeGain` の逓減（sqrt）で **2倍速≠2倍手数**
- [ ] 残：MP切れで魔法型が失速・回復型が長く粘る の明示テストは M2 ハーネスで勝率と併せて確認

**成果物** ✅ `engine/scripts/balance.ts`（`deno task balance`）。総当たり×2000seed×2オリエンテーション、勝率マトリクス＋診断（平均勝率・ミラーsanity・三すくみ・振れ幅）。

**検証項目（企画書の設計が創発しているか）**
- [x] **三すくみが数字から創発** → **3/3 成立**（SPD>MAG>TNK>SPD）。調整1ラウンド（速度逓減 exp 0.5→0.35・装備速度ペナ緩和・乱数±18%）で 2/3→3/3
- [x] **ミラー sanity**：全アーキ 50% ＝ side/id バイアス無し（エンジンは公平）
- [~] **最強ステが無い**：SPD 100%→82% に緩和（要継続）。**死にステ**：CLR 20%・SLP 0% が残課題
- [ ] **勝率ターゲット（100%を作らない・ソフト60:40）**：現状 14/15 がハードカウンター。ソフト化はビルドが近接した時に創発 → **どこまで軟らかくするかの目標設定が必要**
- [!] **スリープの構造的発見**：sleeper が自分の fire で相手を起こす（wakeOnHit）→ 1v1 で sleep がほぼ無価値。**AI/設計判断が必要**（sleep は団体戦のバースト起点向き＝企画書8章と整合）
- [→] 精密なバランス収束は **launch後の live-ops（企画書9章）** が本領。ハーネスはその往復基盤として完成。実データ＋多アーキタイプで継続

**棲み分けの理屈**
- **`engine/`**＝言語・ランタイム独立。純粋関数でDBもUIも知らない → M0〜M2で最初に単独完成する土台。
- **`supabase/`**＝Supabase CLI標準レイアウト。バッチ（`run-battles`）が engine を import して `battle()` を呼ぶだけ。
- **`app/`**＝Flutter標準。engine契約をミラーしたDartモデルを持ち、サーバーが返す eventLog を再生するだけ。

**言語をまたぐ継ぎ目は2箇所だけ**
1. **engine → Edge Function（同一言語・同一ソース）**：engine を Deno製の素のTS（依存ゼロ・ビルド無し）で書き、`functions/run-battles/` から**同じ `battle.ts` を import**。M2ハーネスも Deno（`deno run scripts/balance.ts`）。＝**ハーネスとバッチが1ソース共有＝企画書13.1「1回だけ実装」**。
2. **engine ↔ Flutter（唯一の非対称点）**：TSの型はDartから import 不可。**正本＝`engine/src/schema.ts`**、Flutterは `app/lib/models/` に**対応するDartクラスを手書き**（線を越えるのはJSON＝eventLog、Dartはデシリアライズするだけ）。ドリフトが問題化したら後でコード生成を足す（**今は入れない**）。

**設計メモ（M6.1 で確定）**
- **バッチ分解の粒度**：`roundRobinRounds` が「1ラウンド/日」の単位を返す＝M6.2 の pg_cron は `rounds[day]` を1回ぶん確定→保存→順位更新すればよい（企画書13.5 の定時バッチと自然に一致）
- **冪等はシード設計で担保**：逐次 rng 列（dive 方式）でなく `deriveSeed(seasonSeed, roundKey, matchIndex)` で各カードのシードを独立導出。再実行しても同カードは同シード→同結果。DB 側の status/processed_at は「保存の二重書き」を防ぐだけで、計算は元から冪等
- **タイブレーク公平性**：同勝点は 勝数→敗数少→`deriveSeed(seasonSeed,id)` の順。最後のキーで id アルファベット順の系統的有利を排除（M2 のミラー公平性と同じ精神・テストで検証）
- **引き分けの扱い**：リーグは勝点1ずつ／トーナメントは決着必須ゆえ上位シード進出（決定論）
- **敵/出場者は実キャラのみ**：dive の合成敵と違い、大会は実プレイヤーの `CharacterBuild` を突き合わせる（M6.2 で characters スナップショットを entrants に）
